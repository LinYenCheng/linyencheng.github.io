UrDa 共享機車的使用者體驗有哪些致命錯誤？

從 前端工程師視角 來看，UrDa 共享機車是一款典型的 產品失敗案例分析。其致命錯誤包含：1. 註冊門檻過高：在使用者尚未體驗功能前，即要求填寫過多個資並綁定信用卡；2. APP 使用者體驗 惡劣：UI 元素（如表單格子）過小，且地圖服務竟顯示為「Google Map 開發模式」；3. 網站優化建議 被忽視：Banner 圖片過大（2.5MB）導致載入極慢，嚴重影響 SEO。這類 共享經濟問題 顯示了開發團隊在數位與實體體驗上的脫節。透過這份 產品開發教訓，我們應學會「斷捨離」多餘的個資收集，優先優化核心轉換路徑，避免讓使用者在註冊階段就因系統不穩而流失。

本文將從前端工程師的角度，深入剖析UrDa 共享機車的使用者體驗問題與產品失敗案例。從 APP 註冊流程、網站優化，到實體機車的狀況，揭露一連串「爛到不行」的體驗，希望能為APP 設計與產品開發提供寶貴的借鑒與反思。

註冊畫面

上班族平常生活應該還蠻充實的，上班如果運氣好通常都會很容易進到 Zone 裡面工作，坦白說，其實蠻耗損精神力的，通常下班就會直接進入省電模式，所以一把年紀之後通常? 也開始習慣出去玩不排行程，只求到當地可以找到交通工具。(明明是獨立事件?XDDD)

因為每天都有一萬人降落澎湖馬公，機車已經不太好租，所以決定來試看看共享機車的服務，出發前兩天的晚上，邊電話討論相關事情，一邊嘗試註冊，結果我試了十幾分鐘都一直失敗，還刪掉 APP 重來兩次，直接在電話先靠北兩句:

連前端工程師都沒辦法三分鐘內註冊成功，到底有幾個人可以?

來看一下我失敗的原因，APP 的畫面裡面底下有兩顆社群的按鈕，以我過去實作的經驗來說，這個就是註冊兼登入用的按鈕，等到第一次進入介面使用到關鍵流程時，才需要補齊相關資料，但竟然沒辦法快速登入，過程中失敗了也沒有任何提示。

容易讓工程師誤解的介面，社群按鈕有時候會兼註冊 && 登入

為什麼會想要只先使用快速登入，因為我點了一般註冊，發現想要進去試用介面、查看站點等功能就要填一堆個資 (包含登錄信用卡)，又是一家看起來老闆只想賺錢但沒錢請工程師，介面寫的不怎麼樣且沒聽過的公司，覺得有點危險。

  • 只想體驗介面操作評估工程團隊能力
  • 想了解是否有足夠的機車可租

這個必填真的超多

我會建議拿掉非必要只是為了行銷的資料，還有為什麼要一開始就填信用卡，超怕被盜刷，然後社群按鈕請補上未綁定不能使用的警語出現，這個我小時候就做過，前兩年也做過一次完全不難，工程師請認真一點。

網站體驗

試了第三次之後，決定放棄直接用網頁介面 Orz 我後來是用網頁介面註冊成功，才知道是要先註冊後才能綁定社群帳號，網頁看起來還是外包的，曠 X 國 X 有限公司負責的，看起來整個公司是沒錢請工程師的概念 QQ 不過不愧是外包的，使用起來沒有太大問題，唯一覺得怪怪還可以再進步的就是:

  • Header 選單，Hover 的效果好像沒有對齊，寬度也不一致
  • Banner 圖片品質太高 2.5MB 覺得只是 Banner 不是寫真集應該沒這個必要?
  • 另外一張說明直接上圖很偷懶 1.3MB，無障礙網頁來說這樣真的母湯?

SEO 光是把網站速度提升就可以多不知道多少流量了，這麼簡單的改善，不做嗎? TinyPng 讓無損壓縮變很簡單，用一下，好嗎 XDDD 喔靠，差點忘記這間公司可能沒錢請工程師了 QQ 網站雖然第二次之後就會快取，App 部份我不熟看起來好像不會，還是建議一下圖片剛好就好。

過大的圖片

App 使用體驗

打開 APP 系統後接的地圖有兩種，一看就還在改版，然後手滑上錯了吧?

  • OSM: 要套就支持開源一次全套，前端函式庫有 leaflet、Openlayers 使用上都很簡單，不會寫 APP 可以用 WebView 啊
  • Google Map (竟然還是 develop mode): 沒錢請工程師至少也要花錢啟用服務吧，Google 地圖蠻便宜的阿 QQ

令人傻眼的 Develop Mode

上次真的有在學怎麼寫 App 已經是大學跟研究所了，所以只能列一些體驗上覺得可以進步的地方:

  • 註冊的表單格子有夠小
    • 直接附上設計相關建議
  • APP 第一次進去的時候竟然沒做導覽功能
  • 選單每個按下去後沒有直接反應，需要等待許久
  • 瀏覽站點的圖片用太大解析度，載入圖到可以操作至少超過三秒
  • 綁定信用卡後不知道要怎麼刪除

格子又小又多真的很有壓力

App 的部分應該是沒有買業配文，真的很難 google 到 APP 的使用說明跟體驗心得，建議在改善之後，可以找當地的民宿或部落客合作。

實體機車體驗

貼紙的標示這麼大版面，結果也不留一下 line 的即時客服 QR code 或是疑難排解 FAQ，都已經在車上了還需要這些說明嗎?

覺得少了公司表現最好的部分

這個公司唯一推薦是 Line 的即時客服，只有在場站的時候有加入的 QR code 可以掃，真的很還好我有先加入，解決問題跟回覆的速度都很快，真的很感謝你們。

那個 QR code 請大家一定要記得加

車子本身如果沒有遇到什麼問題，騎乘上還算不錯，雙載的話催到底就是 60 左右，安全帽都有附帽套很棒。

場站密度的部分以我的步行尺度覺得算夠多了，大部分都在知名景點附近，第一次到澎湖自由行的話覺得還算可以，如果只有騎短暫的時間算是非常推薦。

但，還是會遇到有點問題的車子，就讓我們來看下去，第一次租車就拋錨，最後也沒給我們任何 coupon 補貼，只有說到修好為止不算錢，浪費了我跟同學的一小時，這部份真的有點靠北。

真的是傻眼

然後就是出門也要當帝爸狗的生活，大家要學會看燈號變化，敘述情境的時候盡可能把時間、操作步驟都列清楚，對方的客服跟工程師才有辦法查詢或還原真相。

燈號閃爍的方式算是關鍵

最後我們成功遇到了兩種問題，且從頭到尾都沒進行修正

  • 車子油門線故障，造成重啟後騎一小段就會被鎖死，維修人員來之後就是一直幫我們搖晃油門線，晃到正常為止，說如果要修理的話要去機車行調一下油門間隙，因為他沒有帶工具來，請我們自己去機車行，所以來這裡是單純來哈囉的部分? 而且，乾，我們要怎麼去? 我真的差點說 X 你 X，不過職場的歷練讓我展現成熟度，竟然回答出好的，我理解，那客服或是你這邊還能怎麼幫我們? 這裡是荒郊野外，這台車目前就是不能騎，你等一下又要急著趕回台灣?

    運氣很好，維修人員就是大力搖晃到可以暫時騎一段時間，希望不要有可憐的孩子騎到這台

  • 坐墊按鈕故障，撐到附近場站換車之後換到一個較新電池較大的車，但按鈕故障，不過因為按鈕只是觸發連桿，維修人員有提供手動開啟的方式，因為維修人員要準備回台灣了 Orz (聽起來是兩地跑，顧個當地機車行兼職人員不好嗎? 喔，差點忘記這間公司沒錢請工程師，所以很可能也沒錢請維修人員 QQ

    外面按鈕其實裝飾用，重點是驅動底下的連桿，所以也可以用手拉

如果認為自己不會遇到車王，個人是還蠻推薦的，記得租車的話距離不遠可以租電量一半以下的，至少代表有人騎過 XDDD

FAQ：UrDa 共享機車產品體驗常見問題

Q1：為什麼開發人員會誤將地圖設為「Develop Mode」就上架？

A：這通常是 APP 使用者體驗 管理中的低級失誤。在開發階段，開發者可能使用未綁定信用卡的測試 Key。如果上架前未進行完整的冒煙測試（Smoke Test），且沒有 CI/CD 自動化檢查流程，就會導致正式版 APP 呼叫 API 時出現「僅供開發用途」的浮水印，這會極大降低品牌的專業度。

Q2：共享經濟產品在註冊流程中，如何平衡「資安」與「便利性」？

A：高品質的 產品開發教訓 告訴我們：不應在使用者產生黏著度前索取過多個資。建議採用「漸進式註冊」：允許使用者先以社群帳號快速登入查看站點，僅在真正需要「預約車輛」時才要求填寫信用卡與駕照。過早的要求（如 UrDa 範例）會導致極高的流失率。

Q3：實體產品（機車）故障時，數位客服能起到什麼作用？

A：在 共享經濟問題 中，數位與實體是不可分割的。雖然 APP 可能設計得不好，但像 UrDa 提供的 Line 即時客服 若能快速回應並指導「手動排除」（如搖晃油門線或手動開坐墊），能稍微挽回崩潰的使用者體驗。高品質的產品應將 FAQ 與即時客服 QR code 顯眼地標示在設備上，而非藏在繁瑣的 APP 選單中。
