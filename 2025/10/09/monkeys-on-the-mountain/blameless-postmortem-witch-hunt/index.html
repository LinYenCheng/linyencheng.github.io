為什麼抓戰犯是團隊最高昂的隱形成本？

在 職場觀察 中，當系統反覆崩潰，最錯誤的反應是進入 獵巫時刻。這種以「誰的錯」為核心的爭辯會形成 甩鍋閉環：程序猿怪測試，測試員怪時程，管理者怪技術。這種氛圍不僅摧毀信任，更掩蓋了真正的 流程漏洞。解決方案在於導入 無指責檢討會 (Blameless Post-mortem)：假設每個人都已盡力，將矛頭對準系統工具。唯有停止抓戰犯並投資自動化測試，才能徹底消滅救火循環。


無指責檢討會：當團隊不再害怕被指責，隱藏的真相才會浮現，從而將事故轉化為系統演化的養分。

定義：什麼是「無指責檢討會 (Blameless Post-mortem)」？

無指責檢討會是一種事故分析方法，其核心原則是：在不處罰個人的前提下，詳細調查事故的技術與流程根源。其目的是將「責任」轉化為「學習」，防止同類問題再次發生。它假設系統的設計應具備容錯性，而非依賴人的「細心」。

智慧蕉倉系統第五次崩潰了。評審猿怒氣值爆表，管理猿緊急召集會議，氣氛比冰窖還冷。

寓言分析：獵巫時刻：當「誰的錯」比「哪裡錯」更重要

會議室裡，一場精彩的「甩鍋大賽」正式拉開序幕：

  • 管理猿：「程序猿！你的 Code 怎麼回事？為什麼同樣的 Bug 一直出包？！」
  • 程序猿：「我…我上次明明修復了！這肯定是部署環境有問題。測試猿！你怎麼自動化測試沒測出來？」
  • 測試猿：「我們測了啊！但這個 Bug 的觸發條件很極端，而且每次上線的時程都那麼趕，我們根本沒時間做完整的場景覆蓋！這難道不是管理猿你壓的時間太死嗎？」

程序猿怪測試員，測試員怪流程，流程指向管理猿的壓力。一場完美的「甩鍋閉環」形成了。整個下午浪費在爭辯「是誰的錯」，卻沒猴問：「是什麼錯了」？

角色透視：一個關鍵轉變：無指責的魔力

在僵局中，一位資深猴子提議：

「我們能不能先假設：每隻猴子在當下都已經盡了最大努力，且沒有猴子是故意要搞爛系統的？」

這句話開啟了「無指責檢討會」。當大家不再需要為自己辯護，隱藏在冰山下的真相才浮現：

  1. 流程漏洞：團隊根本沒有自動化回歸測試，每次上線都像在賭博。
  2. 工具缺失：目前的部署系統無法自動觸發整合測試，全靠人工手動。
  3. 文化壓力：追求 KPI 的文化讓大家在心理上犧牲了品質來換取速度。

行動指引：消滅未來的問題

抓戰犯是最愚蠢的時間浪費。一場有效的「無指責檢討會」可能只要兩小時，但它建立的自動化流程將消滅未來數百小時的救火成本。領導者的責任是放下指向個人的手指，拿起修補流程的工具。

FAQ：如何落地「無指責文化」？

Q1：在指責風氣強的團隊，如何開啟第一場會議？

A：由「權力最高者」率先示範。主管主動承認自己在資源配置或溝通上的失誤，並明確宣佈：今天的真話不會成為績效考核的負評。這能迅速降低團隊防衛心。

Q2：如果不指責個人，如何確保同樣錯誤不再發生？

A：將問責對象從人移至「代碼與流程」。與其要求某猴細心點（意志力最不可靠），不如要求「增加一道自動化測試」或「修改 API 檢核邏輯」。用技術手段鎖定錯誤。

Q3：習慣性擺爛的人會躲在無指責文化下嗎？

A：不會。無指責是針對「系統故障」而非「行為操守」。對於持續表現不佳者應透過 1-on-1 處理；但在處理技術事故時必須保持無指責，否則關鍵風險資訊會被隱瞞。

Q4：檢討會後產出的 Action Items 沒人做怎麼辦？

A：將 Action Items 列入下個衝刺的「核心任務」。如果修補漏洞不被排入神聖的開發時程，那這場會議就只是另一場表演。

結論：放下手指，拿起鏡子

別讓抓戰犯的熱情，燒掉了團隊的未來。你是在解決問題，還是在尋找替罪羊？真正的強大，是敢於在鏡子中審視系統的缺陷。
